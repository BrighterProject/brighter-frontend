# SEO Optimization — Design Spec

**Date:** 2026-07-21
**Repo:** brighter-frontend
**Branch:** feat/seo-optimization
**Goal:** Bring organic search visibility to parity with competitors (Booking.com, Airbnb, local platforms). Listings first, then programmatic city/region landing pages.

## Context

The frontend is a **TanStack Start** app (React 19) served by a **Nitro SSR server**, so pages are already server-rendered — a sound SEO foundation. i18n is via **intlayer** with `mode: "prefix-all"`, two locales: `bg` (default) and `en`. Every locale carries an explicit prefix (`/bg`, `/en`); `/` redirects to `/bg`. Production host: `pochivka-na-moreto.bg`.

### Current gaps (baseline)

1. **Listing pages are effectively invisible.** `routes/{-$locale}/properties/$propertyId/index.tsx` has no SSR `loader`; it renders `"Loading property..."` server-side and fetches client-side via `useProperty()`. Its `head()` returns a **static generic intlayer string**, so every listing ships an identical `<title>` and meta description.
2. **Root default title is `"TanStack Start Starter"`** (boilerplate) in `__root.tsx`.
3. **OG/Twitter tags exist but are broken** in `routes/{-$locale}/route.tsx`: `og:url` points to `https://площадка.бг/...` (leftover Ploshtadka domain) and `og:image` is a random giphy GIF.
4. **No canonical URLs** — filter params (`?city=&checkIn=…`) create infinite duplicate URLs.
5. **No `hreflang`** alternates despite bg/en.
6. **No JSON-LD** structured data.
7. **No `sitemap.xml`, no `robots.txt`, no `public/` dir.**

## Approach

Centralized **`buildSeo()`** helper consumed by every route `head()`, rolled out in three phases (listings first). Rejected alternatives: per-route inline meta (duplication/drift) and static pre-render/SSG (listings change constantly; runtime SSR + caching is fresher and already available).

Base URL comes from a `VITE_SITE_URL` env var — never hardcoded.

## Phase 1 — Technical fundamentals

### 1.1 `src/lib/seo.ts` — `buildSeo()`
Single source of truth for head metadata. Signature (shape, not final types):

```
buildSeo({
  locale,            // "bg" | "en"
  path,              // locale-less path, e.g. "/properties/123"
  title,             // full <title>
  description,
  image?,            // absolute or path; helper absolutizes via VITE_SITE_URL
  type?,             // "website" | "article" (default website)
  jsonLd?,           // object | object[] -> <script type="application/ld+json">
  noindex?,          // emits robots noindex (auth pages, booking flow)
}) => { meta: [...], links: [...], scripts: [...] }
```

Emits, with **absolute** URLs built from `VITE_SITE_URL`:
- `<title>` + `meta[name=description]`
- `link[rel=canonical]` → `{SITE}/{locale}{path}`
- `link[rel=alternate hreflang=bg|en]` for each locale + `hreflang=x-default` → bg
- OG: `og:title`, `og:description`, `og:image`, `og:url`, `og:type`, `og:site_name`, `og:locale`
- Twitter: `summary_large_image`, title, description, image
- optional `meta[name=robots]=noindex,nofollow`
- optional JSON-LD script(s)

### 1.2 Fix root & locale-layout heads
- `__root.tsx`: replace `"TanStack Start Starter"` with a localized brand default.
- `route.tsx` (`{-$locale}`): route OG through `buildSeo()`; delete площадка.бг and giphy values. Default OG image = a real brand asset in `public/`.

### 1.3 `robots.txt` + dynamic `sitemap.xml` (Nitro server routes)
- `robots.txt`: allow crawl, disallow `/*/auth/*`, `/*/bookings/*`, `/*/checkin/*`, booking flow; point to sitemap.
- `sitemap.xml`: generated by a Nitro server route. Includes static pages, **all active listing IDs** (fetched from properties-ms), and Phase 3 landing pages. Each URL emits per-locale entries with `xhtml:link rel="alternate" hreflang`. Cached (e.g. 1h) to avoid hammering properties-ms.

### 1.4 Canonical on filter/search pages
`/properties` canonicalizes to the param-less locale URL so `?checkIn`/`?city` variants consolidate. (Optionally keep city-filtered canonical self-referential once Phase 3 city pages exist — filtered listing views should canonical to the Phase 3 landing page where one exists.)

## Phase 2 — Dynamic listing pages (highest value)

### 2.1 SSR loader on the listing route
Add a `loader` to `routes/{-$locale}/properties/$propertyId/index.tsx` that fetches the property from properties-ms **server-side** (`GET /properties/{id}?lang={locale}` — public, cacheable, no auth). The loader seeds TanStack Query so `useProperty()` hydrates without a second fetch. Full listing HTML ships in the initial response. On 404, return a proper not-found (with `noindex`).

### 2.2 Dynamic per-listing `head()` via `buildSeo()`
Built from loader data:
- **Title:** `{name} — {property_type} · {city} | Pochivka na Moreto` (localized, unique).
- **Description:** derived from translated description + key facts (guests, bedrooms, city), truncated to ~155 chars.
- **OG image:** the property's real thumbnail (absolutized).
- **Canonical + hreflang** to the bg/en variant of this exact listing.

### 2.3 JSON-LD on listings
Inject via `buildSeo({ jsonLd })`:
- `LodgingBusiness` (or `Hotel`): `name`, `description`, `image[]`, `address` (city, country BG), `geo` (lat/lng), `amenityFeature[]`, `priceRange` from `price_from`, `starRating`/`aggregateRating` from `rating` + `total_reviews` (omit if `total_reviews == 0`).
- `Offer` with `priceCurrency: EUR`, `price` from `price_from`, availability.
- `BreadcrumbList`: Home → Region → City → Listing.

## Phase 3 — Programmatic region→city landing pages

### 3.1 Routes
- Region: `/{-$locale}/accommodation/{regionSlug}` (e.g. `/bg/accommodation/burgas`)
- City:   `/{-$locale}/accommodation/{regionSlug}/{citySlug}` (e.g. `/bg/accommodation/burgas/nesebar`)

SSR'd. Region pages list child cities + counts and link down; city pages list that city's active listings and link into individual listing pages. Localized intro copy per page. Breadcrumbs + `ItemList` (or `CollectionPage`) JSON-LD. Canonical self-referential; hreflang bg/en. Included in sitemap.

Filtered `/properties?region_code=…` / `?city=…` views canonical to the matching Phase 3 landing page.

### 3.2 Backend dependency (properties-ms)
New public endpoint returning the **region → city taxonomy with active-listing counts**, derived from existing `region_code`, `city`, `settlement_ekatte` fields. Needed to (a) render landing pages, (b) enumerate them in the sitemap, (c) build region/city slug ↔ code mapping. Slugs are stable, ASCII-transliterated from names; mapping table owned by properties-ms (or a shared constant) so frontend and sitemap agree.

## Non-goals / YAGNI
- `ru` locale (not configured in intlayer; out of scope).
- AMP.
- Third-party SEO SaaS.
- Review-content generation (uses existing `rating`/`total_reviews` only).

## Testing
- Unit: `buildSeo()` output (canonical/hreflang/OG/JSON-LD correctness, absolutization, noindex) — vitest.
- Unit: JSON-LD builder from a property fixture (valid schema.org shape, rating omitted when 0 reviews).
- Integration: listing route loader seeds query + renders title/description/JSON-LD server-side (jsdom/`renderToString`).
- Manual: validate a sample listing against Google Rich Results Test; verify sitemap.xml + robots.txt.

## Rollout order
Phase 1 → Phase 2 → (backend taxonomy endpoint) → Phase 3. Each phase is independently shippable and measurable via Search Console indexing.
