# SEO Optimization — Design Spec

**Date:** 2026-07-21
**Repo:** brighter-frontend
**Branch:** feat/seo-optimization
**Goal:** Bring organic search visibility to parity with competitors (Booking.com, Airbnb, local platforms). Listings first, then programmatic city/region landing pages.

## Context

The frontend is a **TanStack Start** app (React 19) served by a **Nitro SSR server**, so pages are already server-rendered — a sound SEO foundation. i18n is via **intlayer** with `mode: "prefix-all"`, two locales: `bg` (default) and `en`. Every locale carries an explicit prefix (`/bg`, `/en`); `/` redirects to `/bg`. Production host: `pochivka-na-moreto.bg`.

### Current gaps (baseline)

1. **Listing pages are effectively invisible.** `routes/{-$locale}/properties/$propertyId/index.tsx` has no SSR `loader`; it renders `"Loading property..."` server-side and fetches client-side via `useProperty()`. Its `head()` returns a **static generic intlayer string**, so every listing ships an identical `<title>` and meta description.
2. **Root default title is `"TanStack Start Starter"`** (boilerplate) in `__root.tsx`.
3. **OG/Twitter tags exist but are broken** in `routes/{-$locale}/route.tsx`: `og:url` points to `https://площадка.бг/...` (leftover Ploshtadka domain) and `og:image` is a random giphy GIF.
4. **No canonical URLs** — filter params (`?city=&checkIn=…`) create infinite duplicate URLs.
5. **No `hreflang`** alternates despite bg/en.
6. **No JSON-LD** structured data.
7. **No `sitemap.xml`, no `robots.txt`, no `public/` dir.**

## Approach

Centralized **`buildSeo()`** helper consumed by every route `head()`, rolled out in three phases (listings first). Rejected alternatives: per-route inline meta (duplication/drift) and static pre-render/SSG (listings change constantly; runtime SSR + caching is fresher and already available).

Base URL comes from a `VITE_SITE_URL` env var — never hardcoded.

## Phase 1 — Technical fundamentals

### 1.1 `src/lib/seo.ts` — `buildSeo()`
Single source of truth for head metadata. Signature (shape, not final types):

```
buildSeo({
  locale,            // "bg" | "en"
  path,              // locale-less path, e.g. "/properties/123"
  title,             // full <title>
  description,
  image?,            // absolute or path; helper absolutizes via VITE_SITE_URL
  type?,             // "website" | "article" (default website)
  jsonLd?,           // object | object[] -> <script type="application/ld+json">
  noindex?,          // emits robots noindex (auth pages, booking flow)
}) => { meta: [...], links: [...], scripts: [...] }
```

Emits, with **absolute** URLs built from `VITE_SITE_URL`:
- `<title>` + `meta[name=description]`
- `link[rel=canonical]` → `{SITE}/{locale}{path}`
- `link[rel=alternate hreflang=bg|en]` for each locale + `hreflang=x-default` → bg
- OG: `og:title`, `og:description`, `og:image`, `og:url`, `og:type`, `og:site_name`, `og:locale`
- Twitter: `summary_large_image`, title, description, image
- optional `meta[name=robots]=noindex,nofollow`
- optional JSON-LD script(s)

### 1.2 Fix root & locale-layout heads
- `__root.tsx`: replace `"TanStack Start Starter"` with a localized brand default.
- `route.tsx` (`{-$locale}`): route OG through `buildSeo()`; delete площадка.бг and giphy values. Default OG image = a real brand asset in `public/`.

### 1.3 `robots.txt` + dynamic `sitemap.xml` (Nitro server routes)
- `robots.txt`: allow crawl, disallow `/*/auth/*`, `/*/bookings/*`, `/*/checkin/*`, booking flow; point to sitemap.
- `sitemap.xml`: generated by a Nitro server route. Includes static pages, **all active listing IDs** (fetched from properties-ms), and Phase 3 landing pages. Each URL emits per-locale entries with `xhtml:link rel="alternate" hreflang`. Cached (e.g. 1h) to avoid hammering properties-ms.

### 1.4 Canonical on filter/search pages
`/properties` canonicalizes to the param-less locale URL so `?checkIn`/`?city` variants consolidate. Once Phase 3 city pages exist, a `?city=`/`?region_code=` filtered view canonicalizes to the matching Phase 3 landing page.

### 1.5 Trailing-slash policy
Enforce a single canonical form — **no trailing slash** — via a 301 redirect at the Nitro layer, so `/bg/properties/123/` → `/bg/properties/123` *before* canonical resolution. Prevents duplicate crawls that the canonical tag alone can't pre-empt.

### 1.6 Crawlable pagination (deferred to where lists are paginated)
The current `/properties` list uses **infinite scroll** (`useInfiniteProperties` + `IntersectionObserver`), so pages 2+ have **no crawlable URL** — a crawler only ever sees page 1. This is acceptable for `/properties` (a filter surface that canonicalizes away), but the **Phase 3 landing pages must expose real `?page=N` URLs** with **self-referential canonicals per page** (current Google guidance; `rel=next/prev` was deprecated as an indexing signal in 2019 and is at most an optional Bing nicety, not the mechanism). `buildSeo()` accepts an optional `page` to render the self-canonical correctly.

## Phase 2 — Dynamic listing pages (highest value)

### 2.1 SSR loader on the listing route
Add a `loader` to `routes/{-$locale}/properties/$propertyId/index.tsx` that fetches the property from properties-ms **server-side** (`GET /properties/{id}?lang={locale}` — public, cacheable, no auth). The loader seeds TanStack Query so `useProperty()` hydrates without a second fetch. Full listing HTML ships in the initial response.

**Status handling for dead listings:**
- **Unknown ID** → `404 Not Found` + `noindex` not-found component.
- **Previously public, now delisted** → prefer `410 Gone` (stronger removal signal to Google than 404). *Backend dependency:* this requires `properties-ms` to distinguish "never existed" from "was public, now inactive". Today's public detail endpoint is not confirmed to expose that. **Ship 404+noindex first; upgrade to 410 once the backend signals a gone/delisted state.**

### 2.2 Dynamic per-listing `head()` via `buildSeo()`
Built from loader data:
- **Title:** `{name} | {property_type} in {city} | Pochivka` (localized, unique). `buildSeo()` enforces a **~60-char budget**: truncate `{name}` with an ellipsis first, then drop the `| Pochivka` brand suffix if still over — so `{type} in {city}` (the high-value search terms) always survives. Location is placed before the brand deliberately, matching "[type] in [city]" search intent.
- **Description:** derived from translated description + key facts (guests, bedrooms, city), truncated to ~155 chars.
- **OG image:** the property's real thumbnail (absolutized).
- **Canonical + hreflang** to the bg/en variant of this exact listing.

Use `resolveTranslation(translations, locale)` (from `features/Properties/api/types.ts`) for name/description with bg fallback.

### 2.3 JSON-LD on listings
Inject via `buildSeo({ jsonLd })`:
- `LodgingBusiness` (or `Hotel`): `name`, `description`, `image[]`, `address` (city, country BG), `geo` (lat/lng), `amenityFeature[]`, `priceRange` from `price_from`, `starRating`/`aggregateRating` from `rating` + `total_reviews`.
- `Offer` with `priceCurrency: EUR`, `price` from `price_from`, availability.
- `BreadcrumbList`: Home → Region → City → Listing.

**Structured-data fallbacks (mandatory — invalid partial nodes fail the whole rich result):**
- `price_from` is **nullable**. When null, **omit the `Offer` node and `priceRange` entirely** rather than emit `price: null`/missing `priceCurrency` (which Search Console flags, invalidating the snippet). `has_valid_pricing` gates public visibility so this is rare, but the builder must be defensive.
- When `total_reviews == 0`, **omit `aggregateRating`** (no `ratingValue`/`reviewCount: 0`).
- The JSON-LD builder returns only fully-valid nodes; a missing field drops its node, never the whole graph.

## Phase 3 — Programmatic region→city landing pages

### 3.1 Routes
- Region: `/{-$locale}/accommodation/{regionSlug}` (e.g. `/bg/accommodation/burgas`)
- City:   `/{-$locale}/accommodation/{regionSlug}/{citySlug}` (e.g. `/bg/accommodation/burgas/nesebar`)

SSR'd. Region pages list child cities + counts and link down; city pages expose real `?page=N` paginated URLs (see 1.6) listing that city's active listings and linking into individual listing pages. Localized intro copy per page. Breadcrumbs + `ItemList` (or `CollectionPage`) JSON-LD. Canonical self-referential (per page); hreflang bg/en. Included in sitemap.

Filtered `/properties?region_code=…` / `?city=…` views canonical to the matching Phase 3 landing page.

**URL slug scheme (decided): ASCII, single shared slug per place across locales.**
`/bg/accommodation/burgas/sunny-beach` and `/en/accommodation/burgas/sunny-beach` share the same slug — only the locale prefix differs. Rationale: native Cyrillic paths get percent-encoded in the wild (broken shares/copy-paste) with no ranking benefit; a shared slug keeps hreflang/canonical trivial (same path, differing prefix). Rejected alternative: per-locale slugs (`sunny-beach` / `slanchev-bryag`) — doubles the taxonomy and complicates hreflang for marginal BG-language relevance. *(Revisit only if BG-language slug relevance proves worth the taxonomy cost.)*

**Zero-inventory pages:** when a region/city currently has **0 active listings**, do **not** 404 (preserves accrued backlinks/authority). Render `200` + localized fallback copy ("No active properties in {city} right now — see popular stays nearby in {region}") with links to sibling cities, **plus `noindex`** while empty to avoid thin/doorway-content penalties. Drop the `noindex` automatically once inventory returns.

### 3.2 Backend dependency (properties-ms)
New/extended **public** endpoint returning the **region → city taxonomy with active-listing counts**, derived from existing `region_code`, `city`, `settlement_ekatte` fields. Needed to (a) render landing pages, (b) enumerate them in the sitemap, (c) map region/city slug ↔ code. **Slug derivation and the slug↔code mapping live in `properties-ms`** (a lookup table / materialized mapping), not a frontend constant — the backend already owns the geo fields, and the sitemap generator iterates the endpoint response directly instead of cross-referencing a frontend constant against inventory. Note: a `regions.py` router already exists in properties-ms, so this may extend an existing endpoint rather than be net-new.

## Non-goals / YAGNI
- `ru` locale (not configured in intlayer; out of scope).
- AMP.
- Third-party SEO SaaS.
- Review-content generation (uses existing `rating`/`total_reviews` only).

## Testing
- Unit: `buildSeo()` output (canonical/hreflang/OG/JSON-LD correctness, absolutization, noindex, self-canonical with `page`).
- Unit: title-budget logic — long `{name}` truncates with ellipsis while `{type} in {city}` survives; brand suffix dropped only when still over budget.
- Unit: JSON-LD builder from property fixtures — `aggregateRating` omitted when 0 reviews; `Offer`/`priceRange` omitted when `price_from` is null; remaining nodes stay valid.
- Integration: listing route loader seeds query + renders title/description/JSON-LD server-side (jsdom/`renderToString`); unknown ID → 404+noindex.
- Integration/manual: zero-inventory Phase 3 page returns 200 + fallback + `noindex`; trailing-slash URL 301s to the no-slash form.
- Manual: validate a sample listing against Google Rich Results Test; verify sitemap.xml + robots.txt.

## Rollout order
Phase 1 → Phase 2 → (backend taxonomy endpoint) → Phase 3. Each phase is independently shippable and measurable via Search Console indexing.
